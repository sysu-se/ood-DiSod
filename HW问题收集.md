## HW 问题收集

列举在HW 1、HW1.1过程里，你所遇到的2~3个通过自己学习已经解决的问题，和2~3个尚未解决的问题与挑战

### 已解决

1. 我的 Sudoku / Game 的职责边界是什么？
	1. **上下文**：在 DESIGN 里明确写到，Sudoku 负责保存数独图和基本 guess，Game 负责在 Sudoku 之上维护 undo/redo 历史。
	2. **解决手段**：把“局面表示”和“游戏管理”拆开：Sudoku 专注于当前棋盘与基础校验，Game 统一负责棋盘状态、撤销和重做。

2. 领域对象如何接入 Svelte，才能保证界面能刷新？
	1. **上下文**：早期尝试直接修改对象内部字段，出现了数据变化但界面不稳定更新的问题。
	2. **解决手段**：采用 Game/Sudoku + store adapter 的方式，所有用户操作走 gameStore 命令接口；每次领域状态变化后，通过 store.set 推送新快照，UI 通过订阅统一刷新。

3. Undo/Redo 的职责应该放在哪里？
	1. **上下文**：一开始撤销重做逻辑分散在 UI 和对象层之间，维护困难。
	2. **解决手段**：统一把历史栈和 undo/redo 操作收敛到 Game 层，Sudoku 只负责局面与基本输入，进行解耦化。

### 未解决

1. 暂停状态仍依赖旧 store，为什么棋盘会一直处于不可交互状态？

	1. **上下文**：在 codex_review 里指出，App 里用的是本地 pauseCount，但棋盘是否可点击还在看旧的 @sudoku/stores/game.gamePaused，两个状态源没有真正连起来。
	2. **尝试解决手段**：尝试梳理 App、Board、Header 的调用链，发现欢迎弹窗关闭时只恢复了本地状态，没有同步旧 store；目前还没彻底统一暂停逻辑。

2. 新旧状态源并存，如何彻底收敛为单一可信源？

	1. **上下文**：棋盘部分已接入新 gameStore，但暂停、开始新局、分享、统计等流程仍引用旧 store。
	2. **尝试解决手段**：还是有不少旧的东西没换完；后面打算继续慢慢改，逐渐从stelve抽离出来但现在还没有完全统一好。

3. 数独业务建模还不完整：如何区分题面 givens 与玩家输入？

	1. **上下文**：当前模型更偏向单一可变 grid，导致 givens 不可编辑、校验职责归位等规则难以自然表达。
	2. **尝试解决手段**：我有试着先在适配层把胜负和冲突判断补上，不过感觉这只是临时补了一下，真正把 givens 和可编辑格子分开这件事还没完成
